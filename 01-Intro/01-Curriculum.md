12-Week Intensive Curriculum


Course Focus: Managing Real-World Software Development Projects
Primary Tool: GitHub (Issues, Projects, Milestones)
Methodology Focus: Agile/Scrum
Capstone Project: CBT Application (Frontend Delivery on Existing Backend)


COURSE DESIGN PRINCIPLES

- This is not a theory course

- Every week produces real project artifacts

- You will manage a live software project

- GitHub is your single source of truth


By Week 12, you will have:

- Managed multiple sprints

- Delivered a frontend MVP

- Built a portfolio-grade project history



WEEK-BY-WEEK CURRICULUM

WEEK 1 — Introduction to Software Project Management
Focus: Understanding project management in the context of software development

You Will Learn;

- What a project is (vs operations)

- Why software projects are different

- The role of Project Manager in tech teams

- Why tools and process matter more than titles


You Will Do this;

- Study real software project failures

- Understand the CBT project at a high level

- Set expectations for the course and capstone


Deliverables

i. Short reflection: “Why software projects fail”

ii. High-level understanding of the CBT system



WEEK 2 — Project Initiation & Project Charter
Focus: How real software projects formally begin

You Will Learn;

- Project initiation process

- Project goals, scope, constraints, assumptions

- Stakeholders in IT projects

What You Will Do;

- Initialize your GitHub repository

- Create project documentation structure

- Write a formal Project Charter

Deliverables

i. Project Charter (docs/project-charter.md)

ii. Stakeholder register

iii. Assumptions & constraints document


WEEK 3 — Understanding the Existing System (Legacy Analysis)
Focus: Managing projects that already have existing systems

You Will Learn;

- What “legacy systems” mean

- How PMs work with incomplete documentation

- Reading APIs from a PM perspective (not coding)

What You Will Do;

- Review the CBT backend repository

- Identify system capabilities and limitations

- Document frontend expectations based on backend reality

Deliverables

i. System understanding document

ii. Identified frontend functional areas

iii. Open clarification issues on GitHub


WEEK 4 — Requirements Gathering & User Analysis
Focus: Turning business needs into software requirements

You Will Learn;

- Functional vs non-functional requirements

- User personas and user journeys

- Scope boundaries in software projects

You Will;

- Define CBT user personas (students, instructors, admins)

- Map user journeys

- Document system requirements

Deliverables

i. User personas document

ii. Requirements specification

iii. Scope boundary definition



WEEK 5 — User Stories, Backlog & Acceptance Criteria
Focus: Breaking requirements into executable work

You Will Learn;

- Writing effective user stories

- Acceptance criteria

- Backlog structure and prioritization

You Will;

- Convert requirements into user stories

- Create GitHub Issues for all stories

- Label and prioritize backlog items

Deliverables

i. 30–40 user stories in GitHub Issues

ii. Acceptance criteria for each story

iii. Organized product backlog



WEEK 6 — Agile & Scrum Foundations
Focus: Understanding Agile and Scrum in practice

You Will Learn;

- Agile values and principles

- Scrum roles, artifacts, and ceremonies

- Sprint-based delivery mindset

You Will;

- Assign Scrum roles (real or simulated)

- Define sprint length and cadence

- Prepare for sprint execution

Deliverables

i. Scrum role assignment document

ii. Sprint framework definition

iii. Definition of Ready & Definition of Done



WEEK 7 — GitHub Projects Setup & Sprint Planning
Focus: Turning GitHub into a project management system

You Will Learn;

- GitHub Issues vs Projects vs Milestones

- Kanban vs Sprint boards

- Sprint planning using real tools

You Will;

- Create a GitHub Project board

- Configure workflow columns

- Plan Sprint 1

- Create Sprint 1 milestone

Deliverables

i. Fully configured GitHub Project board

ii. Sprint 1 backlog

iii. Sprint 1 plan document


WEEK 8 — Sprint 1 Execution & Tracking
Focus: Running your first real sprint

You Will Learn;

- Daily standups (async simulation)

- Progress tracking

- Handling blockers and scope creep

You Will;

- Execute Sprint 1

- Update issues and board daily

- Track blockers and decisions

Deliverables

i. Sprint 1 standup log

ii. Sprint 1 review notes

iii. Sprint 1 retrospective



WEEK 9 — Sprint 2 Execution & Backlog Refinement
Focus: Improving delivery through iteration

You Will Learn;

- Backlog grooming

- Responding to change

- Managing technical dependencies

You Will;

- Refine backlog based on Sprint 1 outcomes

- Plan and execute Sprint 2

- Conduct mid-project evaluation

Deliverables

i. Sprint 2 backlog

ii. Sprint 2 review & retrospective

iii. Updated backlog priorities


WEEK 10 — Sprint 3, Integration & Quality Control
Focus: Driving the project toward completion

You Will Learn;

- Managing delivery pressure

- Quality assurance from a PM perspective

- Release readiness

You Will;

- Execute Sprint 3

- Focus on integration and completion

- Track bugs and unfinished work

Deliverables

i. Sprint 3 report

ii. MVP feature completion list

iii. Bug and issue log



WEEK 11 — Risk Management, Stakeholders & Reporting
Focus: Non-technical responsibilities of a Project Manager

You Will Learn

- Risk identification and mitigation

- Stakeholder communication

- Professional reporting

You Will;

- Create a project risk register

- Write stakeholder progress reports

- Document key decisions and trade-offs

Deliverables

i. Risk register

ii. Stakeholder report

iii. Decision log



WEEK 12 — Project Closure & Portfolio Readiness
Focus: Ending projects properly and selling your experience

You Will Learn;

- Formal project closure

- Lessons learned

- Presenting project experience professionally

You Will;

- Close all remaining issues

- Finalize documentation

- Prepare portfolio-ready project summary

Deliverables

i. Final project report

ii. Clean GitHub project history

iii. Portfolio project description

iv. Project defense presentation
